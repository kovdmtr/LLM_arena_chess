# DECISIONS — LLM Chess Arena

Технические решения и их обоснование (ADR-стиль, кратко). Новые решения добавляй
сверху списка с датой.

## D-001 — Язык/стек: Python
**Дата:** 2026-06-09 · **Статус:** принято
**Контекст:** нужно ядро арены с надёжной шахматной логикой и интеграцией LLM/движка.
**Решение:** Python 3.11+.
**Почему:** `python-chess` — зрелая библиотека (валидация, FEN, PGN, SVG-доска, UCI
к Stockfish из коробки); официальные SDK всех трёх провайдеров; минимум своего кода
для шахматной части.
**Альтернативы:** TypeScript/Node (`chess.js`) — слабее шахматная экосистема,
сложнее интеграция движка. Отклонено.

## D-002 — Интерфейс: веб (FastAPI) поверх ядра
**Дата:** 2026-06-09 · **Статус:** принято
**Контекст:** пользователь выбирает модели и хочет наблюдать партию.
**Решение:** ядро арены отделено от UI; веб на FastAPI + WebSocket для живого
просмотра; доска рендерится сервером (SVG).
**Почему:** разделение позволяет тестировать ядро через фейковых игроков/CLI и
лёгкий фронтенд без тяжёлого SPA.
**Замечание:** ядро строится и тестируется в Phase 0–5 до веба (Phase 6).

## D-003 — Секреты в `.env`, несекретное в `config.yaml`
**Дата:** 2026-06-09 · **Статус:** принято
**Контекст:** требование вынести ключи и настройки провайдеров из кода.
**Решение:** API-ключи только в `.env` (gitignore); каталог моделей, путь к движку,
лимиты — в `config.yaml`, который ссылается на **имена** переменных (`api_key_env`),
а не на значения. Коммитится `.env.example`.
**Почему:** ключи не попадают в VCS/артефакты; конфиг моделей версионируется и
читаем. Загрузка через `pydantic-settings`, fail-fast при отсутствии ключа.
**Правило:** ключи никогда не логируются и не пишутся в `game.json`/PGN/HTML.

## D-004 — `game.json` — единственный источник истины
**Дата:** 2026-06-09 · **Статус:** принято
**Контекст:** нужно хранить ходы, сообщения, рассуждения, ошибки, и порождать PGN/HTML.
**Решение:** один структурированный `game.json` на партию; PGN и HTML **генерируются**
из него; артефакты лежат в `games/<id>/`.
**Почему:** воспроизводимость, единый формат, отсутствие рассинхрона между PGN/HTML/логом.

## D-005 — Валидация ходов через `python-chess`
**Дата:** 2026-06-09 · **Статус:** принято
**Решение:** легальность — только `python-chess` (`legal_moves`, `parse_san`,
`parse_uci`). Ход принимается в SAN или UCI; нераспознанный/нелегальный → ретрай.
**Почему:** не изобретать правила; покрытие краевых случаев (рокировка, взятие на
проходе, превращение, повторения).

## D-006 — Технический проигрыш: 3 нелегальные попытки подряд
**Дата:** 2026-06-09 · **Статус:** принято
**Решение:** на одном ходу до 3 попыток; после каждого нелегального хода — коррекция
(причина + легальные ходы); 3 подряд → `technical_loss`. Счётчик сбрасывается после
успешного хода. Лимит конфигурируем (`illegal_move_retries`).
**Почему:** прямое требование; защищает от зацикливания.

## D-007 — Протокол ответа модели: строгий JSON
**Дата:** 2026-06-09 · **Статус:** принято
**Решение:** просим `{ reasoning, move, request_hint, resign }`; парсер устойчив к
тексту вокруг JSON. `move` — SAN или UCI.
**Почему:** надёжно отделяет рассуждение от хода и от запроса подсказки/сдачи;
рассуждение нужно для логов, отчёта и контекста соперника.

## D-008 — Stockfish: опциональная зависимость, обе ★-фичи через него
**Дата:** 2026-06-09 · **Статус:** принято
**Контекст:** выбраны ★-подсказки и ★-пост-анализ.
**Решение:** Stockfish по UCI (`python-chess`) даёт `best_move` (подсказки) и
`evaluate` (анализ). Без бинарника ★-фичи отключаются с предупреждением, база работает.
**Почему:** один движок закрывает обе задачи; опциональность не ломает MVP.

## D-009 — Классификация ходов по centipawn loss + эвристика «блестящий»
**Дата:** 2026-06-09 · **Статус:** принято (реализовано в `analysis/`)
**Решение:** пост-анализ считает centipawn loss относительно лучшего хода (с точки
зрения ходившей стороны); классы `blunder/mistake/inaccuracy/good/brilliant` по
**конфигурируемым** порогам; `brilliant` — эвристика (лучший ход + жертва /
единственный спасающий), опционально подтверждается LLM-комментарием.
**Почему:** прозрачный, воспроизводимый, настраиваемый метод; пороги вынесены в конфиг,
т.к. universally «правильных» значений нет.
**Реализация (`analysis/classify.py`, `analysis/analyzer.py`):**
- cpl хода = `max(0, evaluate(fen_before) + evaluate(fen_after))`. Обе оценки —
  POV стороны, которая ходит в позиции (как отдаёт `StockfishEngine.evaluate`):
  у `fen_after` очередь соперника, поэтому оценка ходившей стороны = `-evaluate(fen_after)`.
  Отрицательный cpl (шум поиска) приравнивается к 0.
- Пороги (`ClassificationThresholds`, из `config.analysis`) задают нижние границы cpl
  для `inaccuracy/mistake/blunder` (должны возрастать — проверяется в `__post_init__`);
  ниже порога неточности — `good`.
- `brilliant` решается поверх классификации: ход почти лучший (`cpl ≤ brilliant_max_cpl`),
  сторона сохраняет перевес (`≥ brilliant_min_eval_cp`) и жертвует материал
  (эвристика `_is_sacrifice`: после размена на целевом поле баланс материала
  отрицателен). `book` автоматически не присваивается.
- `MoveRecord.engine_eval_cp` хранит оценку **с POV белых** (как eval-бар): для хода
  чёрных — со знаком минус. Терминальные `fen_after` движку не отдаются: мат → решающая
  оценка (`±100000`), пат/ничья → 0.
- Без движка (`EngineUnavailableError`) анализ деградирует: `analyze_game` → `None`,
  ходы не размечаются, артефакты базы валидны (D-008).

## D-010 — Подсказки: 3/игрока, по запросу, через протокол ответа
**Дата:** 2026-06-09 · **Статус:** принято (уточнено при реализации в `GameRunner`)
**Решение:** `request_hint: true` в ответе расходует одну из 3 подсказок (если есть
движок и лимит); на следующем запросе в контекст добавляется лучший ход + оценка;
факт и содержание логируются в `HintRecord`.
**Уточнения реализации (`GameRunner._serve_hint`/`_play_turn`):**
- подсказка **тратится только при фактической выдаче**: нет движка, исчерпан лимит
  или движок отказал (`EngineUnavailableError`) → запрос игнорируется, счётчик
  `hints_used` не растёт, ход обрабатывается обычным порядком (база без Stockfish, D-008);
- **не более одной подсказки на полуход**: после выдачи повторный `request_hint` в
  том же ходу игнорируется (модель не «прожигает» весь бюджет на одной позиции, и
  цикл ретраев остаётся ограниченным);
- выданная подсказка **остаётся в контексте до конца полухода** (в т.ч. при ретраях
  нелегального хода, D-006) и пишется в итоговый `MoveRecord` (`hint_used`/`hint`);
- испускается событие `hint` (`EVENT_HINT`) для трассировки/live-просмотра (Phase 6).
**Почему:** соответствует требованию; явный учёт лимита и полная трассируемость;
трата только при выдаче и предел «одна на ход» делают учёт подсказок честным и
ограничивают цикл.

## D-012 — Заявляемые ничьи: авто-ничья
**Дата:** 2026-06-09 · **Статус:** принято (закрывает OQ-1)
**Решение:** при наступлении 50-ходового правила или троекратного повторения партия
автоматически фиксируется как ничья (`auto_claim_draws: true`), без права модели
заявлять/отклонять ничью. `termination` = `fifty_moves` / `repetition`.
**Почему:** проще и детерминированнее; убирает зависимость от поведения модели в
протоколе. Пятикратное повторение / 75 ходов и так автоматические по правилам.

## D-015 — Null-move (`0000` / `--`) отвергается как ход
**Дата:** 2026-06-09 · **Статус:** принято
**Контекст:** `python-chess` принимает `0000`/`--`/`@@@@` как null-move и в `parse_san`,
и в `parse_uci`, причём без проверки легальности — то есть `parse_move` молча
возвращал «ход» с SAN `--`.
**Решение:** `parse_move` отвергает любой null-move (`not move`) с причиной
«не распознан как ход», единым guard в `_to_parsed` (покрывает обе ветки — SAN и UCI).
**Почему:** «пропуск хода» в обычных шахматах невозможен; модель не должна иметь
способа его сделать, а молчаливый проход исказил бы партию и логи.

## D-013 — Картинки доски в отчёте: inline SVG (PNG — опция)
**Дата:** 2026-06-09 · **Статус:** принято (закрывает OQ-2)
**Решение:** доска в `report.html` рендерится как **inline SVG** (`python-chess`).
PNG-растеризация (`cairosvg`) — необязательная опция конфига на будущее.
**Важно:** формат картинок относится **только к HTML-отчёту** и **не влияет** на
совместимость с chess.com/lichess — те импортируют **текстовый PGN**, картинки не
принимают вовсе. PGN остаётся полностью совместим независимо от SVG/PNG.
**Почему:** inline SVG корректно рендерится во всех современных браузерах, легче и
без доп. системных зависимостей; PNG нужен лишь для PDF-экспорта / писем / старых вьюеров.

## D-014 — CLI: только служебный режим
**Дата:** 2026-06-09 · **Статус:** принято (закрывает OQ-3)
**Решение:** `cli.py` существует как служебная точка входа для тестов/CI и локальных
прогонов `GameRunner`, но **не** является поддерживаемым пользовательским интерфейсом.
Единственный пользовательский UI — веб.
**Почему:** не распыляем усилия на полировку двух интерфейсов; CLI остаётся тонкой
обёрткой над ядром, удобной для e2e-прогонов без браузера.

## D-016 — Состав PGN-тегов и санитайз комментариев-рассуждений
**Дата:** 2026-06-09 · **Статус:** принято
**Решение:** PGN несёт семь обязательных тегов (Event/Site/Date/Round/White/Black/
Result) плюс служебные `Termination`, `WhiteModel`/`BlackModel`, `WhiteProvider`/
`BlackProvider`. В тегах игроков — только `display_name`, `model_id` и `provider`,
никаких ключей (D-003). Рассуждения моделей пишутся комментариями `{...}` к ходам;
перед вставкой текст санитайзится: `{`/`}` → `(`/`)`, переводы строк и повторные
пробелы схлопываются в один пробел.
**Почему:** семь тегов — обязательный PGN-стандарт, его понимают lichess/chess.com;
нестандартные `*Model`/`*Provider` они молча игнорируют, а нам дают прослеживаемость
«кто играл». Санитайз скобок обязателен: незакрытая `{`/`}` в рассуждении иначе
порвала бы структуру комментария и сделала PGN непарсимым.

## D-017 — Anthropic: system отдельно от диалога + prompt caching префикса
**Дата:** 2026-06-09 · **Статус:** принято
**Решение:** в `AnthropicProvider` (Messages API) system-реплики (`role="system"`)
не передаются в `messages` (туда идут только `user`/`assistant`), а собираются —
объединением через `\n\n` — в отдельный параметр `system`. Системный промпт
передаётся как один text-блок с `cache_control: {"type": "ephemeral"}`, что включает
prompt caching статичного префикса. Извлечение ответа — конкатенация текста всех
`text`-блоков `response.content`; пустой набор блоков → `ProviderError`.
**Почему:** Messages API структурно отличается от Chat Completions (system — это
top-level параметр, а не сообщение). Системный промпт (правила + строгий JSON-формат,
D-007) неизменен в течение партии и шлётся на каждом ходе, поэтому кэширование
префикса заметно экономит входные токены. `ephemeral` — единственный публичный тип
кэша; ставится на конец статичной части, чтобы закэшировать весь префикс до неё.

## D-018 — Gemini: system_instruction отдельно + маппинг роли assistant→model
**Дата:** 2026-06-09 · **Статус:** принято
**Решение:** в `GeminiProvider` (`google-genai`, `generate_content`) system-реплики
(`role="system"`) не передаются в `contents`, а собираются — объединением через
`\n\n` — в `system_instruction` параметра `GenerateContentConfig` (как и у Anthropic,
system — top-level параметр). Роль ассистента в `contents` называется `model`
(а не `assistant`), `user` остаётся `user`; каждое сообщение — `{"role", "parts":
[{"text"}]}`. Параметры генерации (`temperature`, `max_tokens`→`max_output_tokens`)
тоже идут в `GenerateContentConfig`. Ответ — `response.text` (конкатенация текстовых
частей); пустой/`None` → `ProviderError`. Без system-реплик `system_instruction=None`.
**Почему:** Gemini API структурно отличается от Chat Completions именованием ролей
и местом системной инструкции. В отличие от Anthropic, отдельного prompt caching
здесь не задаём: в `google-genai` кэширование префикса требует явного explicit-cache
API (`client.caches`), что выходит за рамки этой задачи и может быть добавлено в
Phase 7 («закалка») при необходимости.

## D-019 — Системный промпт на английском, статичен в пределах партии
**Дата:** 2026-06-09 · **Статус:** принято
**Решение:** текст системного промпта (`prompts/system.py`, `build_system_prompt`)
пишется на английском и не зависит от позиции — только от лимитов партии
(`hints_per_player`, `illegal_move_retries`), которые фиксированы на всю игру.
Канонические ключи протокола ответа вынесены в `RESPONSE_KEYS` — единый источник
для текста промпта и тестов; согласованность с парсером проверяется тестом
(`parse_response` разбирает пример-объект из промпта). Язык поля `reasoning` модели
не навязывается.
**Почему:** на английском все три провайдера надёжнее держат строгий JSON-формат
(D-007) и шахматную нотацию. Статичность промпта при неизменных лимитах — условие
prompt caching префикса (D-017): кэш живёт в пределах одной партии, а лимиты в ней
не меняются, поэтому их подстановка кэш не ломает.

## D-020 — Окончание партии: финализация result/termination и resign
**Дата:** 2026-06-09 · **Статус:** принято
**Контекст:** после остановки игрового цикла нужно проставить итог партии в
`GameRecord`, согласовав коды причин с документированным словарём (спека 5.3, D-012),
и обработать добровольную сдачу из протокола ответа (D-007).
**Решение:** `GameRunner._finalize` проставляет `result`/`termination` из
`Board.outcome()` только для **обычных** окончаний (мат/пат/ничьи) и только если
исход ещё не зафиксирован. Технические исходы выставляются в момент события и не
перезаписываются: техническое поражение (D-006) и сдача (`resign: true` →
`termination=resign`, результат = победа соперника). Коды повторения
(`threefold_repetition`/`fivefold_repetition`) сводятся к единому `repetition`
(`_normalize_termination`); прочие коды доски проходят без изменений. Обрыв по
`max_plies` оставляет `result="*"` (партия не окончена). Итоговое событие
`EVENT_GAME_OVER` несёт `result`/`termination` в нагрузке.
**Почему:** единая точка финализации исключает рассинхрон; различие 3-/5-кратного
повторения для пользователя несущественно, поэтому сводится к `repetition` (как в
D-012). Сдача — часть протокола D-007, обрабатывается симметрично техпоражению
(победа соперника), без отдельного механизма «соглашения на ничью» (его нет, D-012).

## D-021 — Список легальных ходов в промпте: опционален, по умолчанию выключен
**Дата:** 2026-06-09 · **Статус:** принято
**Контекст:** раньше контекст хода всегда содержал полный список легальных ходов в
SAN, из которого модель обязана была выбрать. Это сильная подсказка: она снимает с
модели задачу самой удерживать легальность и завышает «силу» игры. Хотелось режима,
где модель ходит по позиции (FEN/PGN) самостоятельно, а движок лишь проверяет ход.
**Решение:** введён флаг `include_legal_moves` (`PlayerSettings` и `config.yaml →
ArenaConfig`), сквозной до `prompts`: `build_context`/`context_message` кладут секцию
`Legal moves (SAN):` только при `True`, а `build_system_prompt`/`system_message`
переключают формулировки (обещание списка, текст коррекции и предупреждение о
страйках). **Дефолт партии (`PlayerSettings`) — `False`**: список не передаётся,
легальность проверяется уже после ответа в `GameRunner._resolve_move`
(`parse_move` по реальной доске) с прежним ретраем нелегального хода (D-006) и
техпоражением на лимите. Дефолт самих функций `prompts` оставлен `True` (обратная
совместимость прямых вызовов/тестов; раннер всегда передаёт значение из настроек).
**Почему:** проверка легальности уже существует и не зависит от подсказки — нелегальный
ход физически не доходит до доски, поэтому убрать список безопасно. Флаг делает смену
обратимой и сравнимой A/B. Компромисс: без списка слабые модели чаще присылают
нелегальные ходы → больше ретраев (стоимость/задержка) и техпоражений; это осознанный
выбор в пользу «честной» игры по позиции, регулируемый флагом.

## D-022 — Повтор вызова провайдера: эвристика транзиентности + экспоненциальный backoff
**Дата:** 2026-06-09 · **Статус:** принято
**Контекст:** LLM-API периодически отвечают временными ошибками (rate-limit 429,
таймаут, обрыв соединения, перегрузка/5xx). Падать на первом же таком сбое для
партии расточительно. Нужна устойчивость, но без бессмысленных повторов постоянных
ошибок (неверный ключ, 400/404) и без жёсткой привязки к классам исключений каждого
SDK (openai/anthropic/google — у всех свои).
**Решение:** слой `providers/retry.py`: `is_transient_error` решает «повторять ли» по
объединению признаков — числовой `status_code/status/code ∈ {429,500,502,503,504}`
**или** маркер в имени класса/тексте ошибки (`rate limit`, `timeout`, `connection`,
`overloaded`, `unavailable`, …); `call_with_retry(fn, RetryConfig, sleep, rng)`
повторяет только транзиентные ошибки до `attempts` попыток с задержкой
`base_delay·multiplier^(n-1)`, обрезанной `max_delay` и размытой `jitter∈[0,1]`.
Интеграция — на уровне базового `LLMProvider._call(fn)`: каждая реализация оборачивает
свой единственный SDK-вызов в `self._call(...)` **внутри** существующего `try/except`,
поэтому итоговое (или сразу постоянное) исключение по-прежнему превращается в
`ProviderError` с маскированием ключа (D-003). Политика берётся из `config.yaml →
retry` (`RetryConfig`), передаётся через `create_provider(retry=...)`; по умолчанию —
3 попытки, экспонента 0.5→8 c, jitter 0.1.
**Почему:** эвристика по строке/коду статуса покрывает все три SDK одним механизмом и
не ломается при обновлении их иерархий исключений. `sleep`/`rng` инъектируются — backoff
детерминирован в тестах без реальных пауз. Повтор внутри `try/except` сохраняет
прежний контракт `complete` (наружу — только `ProviderError`, без утечки ключа).
Компромисс: эвристика может изредка повторить ошибку, которая на деле постоянна (или
наоборот) — цена за независимость от частных классов SDK; границы повторов и задержек
регулируются конфигом.

## D-023 — Логирование: namespaced `arena`-логгер, структурный вывод, маскирование на форматтере
**Дата:** 2026-06-09 · **Статус:** принято
**Контекст:** на закалке нужно наблюдаемое поведение (старт/финал/сбой партии,
ретраи), но без двух рисков: утечки API-ключа в логи (D-003) и захвата корневого
логгера хоста (приложение — ещё и библиотека/пакет). Точечного `mask_secret` в
провайдерах недостаточно: ключ может всплыть в трейсбеке, `extra`-поле, чужом
сообщении.
**Решение:** слой `obs` (`obs/log.py`): все логи под пространством имён `arena`
(`get_logger("web") → arena.web`); `configure_logging` настраивает **только** этот
логгер (один handler, `propagate=False`), идемпотентен (повтор без `force` не
дублирует handler), умеет текст или JSON. Маскирование — на уровне
`StructuredFormatter.format`: итоговая строка (сообщение+args+extra+трейсбек)
пропускается через `redact`, который вырезает значения из **глобального реестра
секретов** (`register_secret(s)`, длинные раньше коротких). Реестр наполняется
автоматически при создании любого провайдера (`LLMProvider.__init__ →
register_secret(model.api_key)`) и из загруженных `Settings` в веб
(`_ensure_settings`). На корне `arena` стоит `NullHandler` — до `configure_logging`
вывода нет (нет «No handlers», тихо в библиотечном режиме).
**Почему:** маскирование на форматтере — единственная точка, через которую проходит
**весь** вывод, поэтому ловит ключ где угодно (в т.ч. в трейсбеке `exc_info`),
в отличие от точечной чистки в местах вызова. Глобальный реестр + авторегистрация
при создании провайдера означает «ключ замаскирован, как только он появился в
системе», без ручного перечисления. Namespaced-логгер с `propagate=False` не мешает
хосту и тестам. Компромисс: маскирование — подстрочная замена, поэтому очень
короткий/общий «секрет» теоретически затрёт легитимный текст; для API-ключей
(длинные уникальные строки) это не проблема, а пустые значения реестр игнорирует.

## D-024 — Единый путь включения движка: `build_engine` (конфиг → движок|None) + движок в веб
**Дата:** 2026-06-09 · **Статус:** принято
**Контекст:** деградация без Stockfish (D-008) реализована у каждого потребителя
по-своему: `GameRunner._serve_hint` молча не выдаёт подсказку, `analyze_game`
возвращает `None`. Но *решение* «движок есть/нет» (проверка `enabled`, запуск
бинарника, обработка `EngineUnavailableError`) нигде не было собрано в одном месте, и
веб-партии вовсе не подключали движок — играли на базовом уровне, без подсказок и без
★-разметки в отчёте.
**Решение:** `engine.build_engine(EngineConfig, *, depth=None, opener=None)` —
единственная точка решения: `enabled=false` → `None`; бинарник не стартует
(`EngineUnavailableError`) → `None`; иначе **открытый** `StockfishEngine` (процесс
поднят сразу — недоступность видна на старте, а не на первом ходу). Вызывающий лишь
проверяет `if engine is not None`. Веб подключил движок через этот путь:
`GameManager` получил `engine_factory` (строит движок-или-`None` на партию),
`analysis_config`, `analysis_depth`; в `_run` движок передаётся в `GameRunner`
(подсказки D-010), после `play()` — `analyze_game` заполняет `record.analysis`
(D-009), в `finally` движок закрывается. `web.app._get_manager` собирает фабрику из
`settings.config.engine` (`build_engine(engine_cfg, depth=hint_depth)`). Один процесс
обслуживает и подсказки (глубина по умолчанию = `hint_depth`), и анализ
(`analyze_game(depth=analysis_depth)`).
**Почему:** сосредоточить «движок есть/нет» в одной функции — значит, что новые
потребители (CLI, будущие сценарии) деградируют единообразно и без копипасты
try/except. Открытие процесса в фабрике делает недоступность ранней и явной.
Per-game фабрика изолирует процесс на партию и гарантирует его закрытие. Веб теперь
даёт ★ подсказки и пост-анализ в отчёте; при отсутствии бинарника всё так же
работает на базовом уровне. LLM-комментарий ключевых моментов (опц., D-009) в веб
пока не подключён — требует провайдера-комментатора; оставлено на потом.

## D-025 — Фича «стратегия»: приватный rolling-план с возвратом самому себе
**Дата:** 2026-06-10 · **Статус:** принято
**Контекст:** модель на каждом ходу получала «срез с нуля» (FEN+PGN+рассуждения) и
не несла памяти о *замысле* — играла как оценщик позиции, а не как игрок, ведущий
партию. Хотелось, чтобы ход подразумевал стратегию, а сама стратегия возвращалась
модели на следующем ходу.
**Решение:** в протокол ответа (D-007) добавлены поля `strategy` (приватный план на
ближайшие ходы) и `plan_status` (`start`/`continue`/`adapt`/`abandon` — отношение к
прошлому плану); `reasoning` остаётся объяснением *этого* хода, `strategy` —
многоходовым замыслом. План пишется в `MoveRecord` (источник истины: «текущий план
стороны» = `strategy` её последнего хода), а `build_context` (под флагом
`include_strategy`) инъектирует **только собственный** план стороны (`_plan_section`)
с просьбой продолжить/скорректировать/отбросить. Решения по развилкам: показываем
**только последний** план (rolling, дёшево по токенам и кэшу D-017); план **приватен**
(соперник его не видит — как в реальной партии); фича **включена по умолчанию**
(`ArenaConfig.strategy.enabled`, зеркало `PlayerSettings.strategy_enabled`,
переключаемо); статус — **структурный** (`continue/adapt/abandon`, для отчёта и
проверки непрерывности). Системный промпт описывает поля только при включённой фиче
(базовый промпт байт-в-байт прежний → кэш и старые тесты целы); `parse_response`
читает поля мягко (неизвестный статус → `start`). Заодно добавлен мост
`ArenaConfig.to_player_settings()` — `config.yaml` теперь реально драйвит настройки
партии во всех точках входа (web/cli/tournament), что раньше не работало.
**Почему:** rolling-план даёт непрерывность замысла при ограниченном росте контекста;
приватность сохраняет характер шахматной партии; структурный статус делает видимым,
когда модель *сознательно* меняет курс (бейдж в отчёте). Хранение плана по-ходно (а не
отдельным состоянием на `GameRecord`) исключает рассинхрон — единственный источник
истины тот же, что и для PGN/отчёта (D-004).
