# DECISIONS — LLM Chess Arena

Технические решения и их обоснование (ADR-стиль, кратко). Новые решения добавляй
сверху списка с датой.

## D-001 — Язык/стек: Python
**Дата:** 2026-06-09 · **Статус:** принято
**Контекст:** нужно ядро арены с надёжной шахматной логикой и интеграцией LLM/движка.
**Решение:** Python 3.11+.
**Почему:** `python-chess` — зрелая библиотека (валидация, FEN, PGN, SVG-доска, UCI
к Stockfish из коробки); официальные SDK всех трёх провайдеров; минимум своего кода
для шахматной части.
**Альтернативы:** TypeScript/Node (`chess.js`) — слабее шахматная экосистема,
сложнее интеграция движка. Отклонено.

## D-002 — Интерфейс: веб (FastAPI) поверх ядра
**Дата:** 2026-06-09 · **Статус:** принято
**Контекст:** пользователь выбирает модели и хочет наблюдать партию.
**Решение:** ядро арены отделено от UI; веб на FastAPI + WebSocket для живого
просмотра; доска рендерится сервером (SVG).
**Почему:** разделение позволяет тестировать ядро через фейковых игроков/CLI и
лёгкий фронтенд без тяжёлого SPA.
**Замечание:** ядро строится и тестируется в Phase 0–5 до веба (Phase 6).

## D-003 — Секреты в `.env`, несекретное в `config.yaml`
**Дата:** 2026-06-09 · **Статус:** принято
**Контекст:** требование вынести ключи и настройки провайдеров из кода.
**Решение:** API-ключи только в `.env` (gitignore); каталог моделей, путь к движку,
лимиты — в `config.yaml`, который ссылается на **имена** переменных (`api_key_env`),
а не на значения. Коммитится `.env.example`.
**Почему:** ключи не попадают в VCS/артефакты; конфиг моделей версионируется и
читаем. Загрузка через `pydantic-settings`, fail-fast при отсутствии ключа.
**Правило:** ключи никогда не логируются и не пишутся в `game.json`/PGN/HTML.

## D-004 — `game.json` — единственный источник истины
**Дата:** 2026-06-09 · **Статус:** принято
**Контекст:** нужно хранить ходы, сообщения, рассуждения, ошибки, и порождать PGN/HTML.
**Решение:** один структурированный `game.json` на партию; PGN и HTML **генерируются**
из него; артефакты лежат в `games/<id>/`.
**Почему:** воспроизводимость, единый формат, отсутствие рассинхрона между PGN/HTML/логом.

## D-005 — Валидация ходов через `python-chess`
**Дата:** 2026-06-09 · **Статус:** принято
**Решение:** легальность — только `python-chess` (`legal_moves`, `parse_san`,
`parse_uci`). Ход принимается в SAN или UCI; нераспознанный/нелегальный → ретрай.
**Почему:** не изобретать правила; покрытие краевых случаев (рокировка, взятие на
проходе, превращение, повторения).

## D-006 — Технический проигрыш: 3 нелегальные попытки подряд
**Дата:** 2026-06-09 · **Статус:** принято
**Решение:** на одном ходу до 3 попыток; после каждого нелегального хода — коррекция
(причина + легальные ходы); 3 подряд → `technical_loss`. Счётчик сбрасывается после
успешного хода. Лимит конфигурируем (`illegal_move_retries`).
**Почему:** прямое требование; защищает от зацикливания.

## D-007 — Протокол ответа модели: строгий JSON
**Дата:** 2026-06-09 · **Статус:** принято
**Решение:** просим `{ reasoning, move, request_hint, resign }`; парсер устойчив к
тексту вокруг JSON. `move` — SAN или UCI.
**Почему:** надёжно отделяет рассуждение от хода и от запроса подсказки/сдачи;
рассуждение нужно для логов, отчёта и контекста соперника.

## D-008 — Stockfish: опциональная зависимость, обе ★-фичи через него
**Дата:** 2026-06-09 · **Статус:** принято
**Контекст:** выбраны ★-подсказки и ★-пост-анализ.
**Решение:** Stockfish по UCI (`python-chess`) даёт `best_move` (подсказки) и
`evaluate` (анализ). Без бинарника ★-фичи отключаются с предупреждением, база работает.
**Почему:** один движок закрывает обе задачи; опциональность не ломает MVP.

## D-009 — Классификация ходов по centipawn loss + эвристика «блестящий»
**Дата:** 2026-06-09 · **Статус:** принято
**Решение:** пост-анализ считает centipawn loss относительно лучшего хода (с точки
зрения ходившей стороны); классы `blunder/mistake/inaccuracy/good/brilliant` по
**конфигурируемым** порогам; `brilliant` — эвристика (лучший ход + жертва /
единственный спасающий), опционально подтверждается LLM-комментарием.
**Почему:** прозрачный, воспроизводимый, настраиваемый метод; пороги вынесены в конфиг,
т.к. universally «правильных» значений нет.

## D-010 — Подсказки: 3/игрока, по запросу, через протокол ответа
**Дата:** 2026-06-09 · **Статус:** принято
**Решение:** `request_hint: true` в ответе расходует одну из 3 подсказок (если есть
движок и лимит); на следующем запросе в контекст добавляется лучший ход + оценка;
факт и содержание логируются в `HintRecord`.
**Почему:** соответствует требованию; явный учёт лимита и полная трассируемость.

## D-011 — Расположение проекта: `Chess_LLM_Claude/`
**Дата:** 2026-06-09 · **Статус:** принято
**Решение:** проект живёт в пустой директории `Chess_LLM_Claude/`.
**Почему:** подходящее имя и чистое место; `cloude_test/` содержит несвязанные файлы
(квиз) и смешивать не нужно.

## D-012 — Заявляемые ничьи: авто-ничья
**Дата:** 2026-06-09 · **Статус:** принято (закрывает OQ-1)
**Решение:** при наступлении 50-ходового правила или троекратного повторения партия
автоматически фиксируется как ничья (`auto_claim_draws: true`), без права модели
заявлять/отклонять ничью. `termination` = `fifty_moves` / `repetition`.
**Почему:** проще и детерминированнее; убирает зависимость от поведения модели в
протоколе. Пятикратное повторение / 75 ходов и так автоматические по правилам.

## D-015 — Null-move (`0000` / `--`) отвергается как ход
**Дата:** 2026-06-09 · **Статус:** принято
**Контекст:** `python-chess` принимает `0000`/`--`/`@@@@` как null-move и в `parse_san`,
и в `parse_uci`, причём без проверки легальности — то есть `parse_move` молча
возвращал «ход» с SAN `--`.
**Решение:** `parse_move` отвергает любой null-move (`not move`) с причиной
«не распознан как ход», единым guard в `_to_parsed` (покрывает обе ветки — SAN и UCI).
**Почему:** «пропуск хода» в обычных шахматах невозможен; модель не должна иметь
способа его сделать, а молчаливый проход исказил бы партию и логи.

## D-013 — Картинки доски в отчёте: inline SVG (PNG — опция)
**Дата:** 2026-06-09 · **Статус:** принято (закрывает OQ-2)
**Решение:** доска в `report.html` рендерится как **inline SVG** (`python-chess`).
PNG-растеризация (`cairosvg`) — необязательная опция конфига на будущее.
**Важно:** формат картинок относится **только к HTML-отчёту** и **не влияет** на
совместимость с chess.com/lichess — те импортируют **текстовый PGN**, картинки не
принимают вовсе. PGN остаётся полностью совместим независимо от SVG/PNG.
**Почему:** inline SVG корректно рендерится во всех современных браузерах, легче и
без доп. системных зависимостей; PNG нужен лишь для PDF-экспорта / писем / старых вьюеров.

## D-014 — CLI: только служебный режим
**Дата:** 2026-06-09 · **Статус:** принято (закрывает OQ-3)
**Решение:** `cli.py` существует как служебная точка входа для тестов/CI и локальных
прогонов `GameRunner`, но **не** является поддерживаемым пользовательским интерфейсом.
Единственный пользовательский UI — веб.
**Почему:** не распыляем усилия на полировку двух интерфейсов; CLI остаётся тонкой
обёрткой над ядром, удобной для e2e-прогонов без браузера.

## D-016 — Состав PGN-тегов и санитайз комментариев-рассуждений
**Дата:** 2026-06-09 · **Статус:** принято
**Решение:** PGN несёт семь обязательных тегов (Event/Site/Date/Round/White/Black/
Result) плюс служебные `Termination`, `WhiteModel`/`BlackModel`, `WhiteProvider`/
`BlackProvider`. В тегах игроков — только `display_name`, `model_id` и `provider`,
никаких ключей (D-003). Рассуждения моделей пишутся комментариями `{...}` к ходам;
перед вставкой текст санитайзится: `{`/`}` → `(`/`)`, переводы строк и повторные
пробелы схлопываются в один пробел.
**Почему:** семь тегов — обязательный PGN-стандарт, его понимают lichess/chess.com;
нестандартные `*Model`/`*Provider` они молча игнорируют, а нам дают прослеживаемость
«кто играл». Санитайз скобок обязателен: незакрытая `{`/`}` в рассуждении иначе
порвала бы структуру комментария и сделала PGN непарсимым.

## D-017 — Anthropic: system отдельно от диалога + prompt caching префикса
**Дата:** 2026-06-09 · **Статус:** принято
**Решение:** в `AnthropicProvider` (Messages API) system-реплики (`role="system"`)
не передаются в `messages` (туда идут только `user`/`assistant`), а собираются —
объединением через `\n\n` — в отдельный параметр `system`. Системный промпт
передаётся как один text-блок с `cache_control: {"type": "ephemeral"}`, что включает
prompt caching статичного префикса. Извлечение ответа — конкатенация текста всех
`text`-блоков `response.content`; пустой набор блоков → `ProviderError`.
**Почему:** Messages API структурно отличается от Chat Completions (system — это
top-level параметр, а не сообщение). Системный промпт (правила + строгий JSON-формат,
D-007) неизменен в течение партии и шлётся на каждом ходе, поэтому кэширование
префикса заметно экономит входные токены. `ephemeral` — единственный публичный тип
кэша; ставится на конец статичной части, чтобы закэшировать весь префикс до неё.

## D-018 — Gemini: system_instruction отдельно + маппинг роли assistant→model
**Дата:** 2026-06-09 · **Статус:** принято
**Решение:** в `GeminiProvider` (`google-genai`, `generate_content`) system-реплики
(`role="system"`) не передаются в `contents`, а собираются — объединением через
`\n\n` — в `system_instruction` параметра `GenerateContentConfig` (как и у Anthropic,
system — top-level параметр). Роль ассистента в `contents` называется `model`
(а не `assistant`), `user` остаётся `user`; каждое сообщение — `{"role", "parts":
[{"text"}]}`. Параметры генерации (`temperature`, `max_tokens`→`max_output_tokens`)
тоже идут в `GenerateContentConfig`. Ответ — `response.text` (конкатенация текстовых
частей); пустой/`None` → `ProviderError`. Без system-реплик `system_instruction=None`.
**Почему:** Gemini API структурно отличается от Chat Completions именованием ролей
и местом системной инструкции. В отличие от Anthropic, отдельного prompt caching
здесь не задаём: в `google-genai` кэширование префикса требует явного explicit-cache
API (`client.caches`), что выходит за рамки этой задачи и может быть добавлено в
Phase 7 («закалка») при необходимости.

## D-019 — Системный промпт на английском, статичен в пределах партии
**Дата:** 2026-06-09 · **Статус:** принято
**Решение:** текст системного промпта (`prompts/system.py`, `build_system_prompt`)
пишется на английском и не зависит от позиции — только от лимитов партии
(`hints_per_player`, `illegal_move_retries`), которые фиксированы на всю игру.
Канонические ключи протокола ответа вынесены в `RESPONSE_KEYS` — единый источник
для текста промпта и тестов; согласованность с парсером проверяется тестом
(`parse_response` разбирает пример-объект из промпта). Язык поля `reasoning` модели
не навязывается.
**Почему:** на английском все три провайдера надёжнее держат строгий JSON-формат
(D-007) и шахматную нотацию. Статичность промпта при неизменных лимитах — условие
prompt caching префикса (D-017): кэш живёт в пределах одной партии, а лимиты в ней
не меняются, поэтому их подстановка кэш не ломает.
