# ROADMAP — LLM Chess Arena

Этапы от каркаса до полной ★-функциональности. Внутри этапа — атомарные задачи
(подробно в `docs/TODO.md`), каждая = один коммит. Веб — выбранный интерфейс, но
ядро строится первым и тестируется через фейковых игроков/CLI, чтобы веб лёг на
готовую логику.

## Phase 0 — Каркас и конфиг
Репозиторий, зависимости, загрузка `config.yaml` + `.env`, каталог моделей.
**Готово, когда:** проект ставится, настройки и каталог моделей читаются, ключи
валидируются (fail-fast), есть `.env.example`.

## Phase 1 — Шахматное ядро (`core`)
Обёртка `python-chess`: FEN, легальные ходы, применение хода, детект окончания;
парсинг хода (SAN/UCI) из текста; сборка PGN из структуры.
**Готово, когда:** можно проиграть партию из списка ходов в коде, получить
корректный PGN, все юнит-тесты зелёные.

## Phase 2 — Провайдеры LLM
Единый интерфейс `LLMProvider` + реализации OpenAI / Anthropic / Gemini; ретраи,
маскирование ключей. `ModelPlayer` поверх провайдера.
**Готово, когда:** каждый провайдер возвращает ответ (на моках в тестах), реальный
вызов работает с ключом; `ModelPlayer` отдаёт распарсенный `LLMResponse`.

## Phase 3 — Игровой цикл (`arena`)
Сборка контекста (FEN, легальные ходы, PGN, история, объяснения обеих сторон),
`GameRunner`: чередование, валидация, ретраи нелегальных ходов, 3-strike → тех.
поражение, детект окончания, запись `GameRecord`.
**Готово, когда:** два фейковых детерминированных игрока доигрывают партию;
нелегальные ходы и техническое поражение отрабатывают; `game.json` сохраняется.

## Phase 4 — Артефакты (PGN + HTML)
Экспорт `game.pgn` (открывается в lichess/chess.com), рендер доски, self-contained
HTML-отчёт с ходами, рассуждениями, итогом.
**Готово, когда:** из `game.json` получаются валидный PGN и читаемый HTML-отчёт.

## Phase 5 — ★ Движок: подсказки и анализ
Stockfish-обёртка (eval, best move); 3 подсказки/партию по протоколу; пост-анализ
centipawn loss и классификация ходов; (опц.) LLM-комментарий ключевых моментов.
**Готово, когда:** модель может запросить подсказку (лимит соблюдается),
после партии ходы размечены классами, сводка в отчёте.

## Phase 6 — ★ Веб-интерфейс
FastAPI: страница выбора моделей из каталога, старт партии, WebSocket-живой
просмотр (доска/ход/рассуждение), список и просмотр отчётов.
**Готово, когда:** партия запускается и наблюдается из браузера, отчёт доступен.

## Phase 7 — Закалка
Ретраи/backoff провайдеров, логирование с маскированием ключей, корректная
деградация без Stockfish, e2e-прогон, пример партии в репозитории, финал доков.
**Готово, когда:** проект устойчив к типовым сбоям, документация актуальна,
есть демонстрационная партия.

## Порядок и зависимости
```
P0 → P1 → P2 → P3 → P4
                 └→ P5 (★, нужен P3)
P4,P5 → P6 (★ веб) → P7
```
MVP без ★ = Phase 0–4. Полная версия = до Phase 7.
