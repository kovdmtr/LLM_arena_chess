# CLAUDE.md — LLM Chess Arena

Этот файл — точка входа для Claude Code и любого разработчика в репозитории.
Читай его первым. Подробности — в `docs/`.

## Что это за проект

**LLM Chess Arena** — арена, где две LLM-модели (OpenAI, Anthropic, Google Gemini)
играют друг против друга в обычные шахматы без контроля времени. Система проверяет
легальность каждого хода, ведёт полный лог (ходы, сообщения, рассуждения моделей,
ошибки), на выходе даёт PGN (для lichess/chess.com) и self-contained HTML-отчёт.

★-фичи: 3 подсказки на партию от Stockfish каждой модели и автоматический пост-анализ
качества ходов (блестящий / хороший / неточность / ошибка / зевок).

## Стек (кратко)

- **Python 3.11+**
- `python-chess` — валидация ходов, FEN, PGN, рендер доски (SVG)
- SDK провайдеров: `openai`, `anthropic`, `google-genai`
- `Stockfish` (UCI) — подсказки и пост-анализ
- `FastAPI` + `Uvicorn` + WebSocket — веб-UI и живой просмотр партии
- `pydantic` / `pydantic-settings` — конфиг и валидация
- `Jinja2` — HTML-отчёт
- `pytest` — тесты

## Ритуал сессии (важно для преемственности между сессиями)

Память между сессиями НЕ сохраняется — состояние реконструируется из файлов и git.
Поэтому строго соблюдай ритуал.

**Автозапуск (чтобы не повторять преамбулу каждую сессию).** Этот файл читается
автоматически в начале каждой сессии. Поэтому как только пользователь подаёт **любой**
сигнал продолжать работу (напр. «продолжай», «дальше», «go», «/resume» — без
необходимости заново перечислять «прочитай CLAUDE.md / PROGRESS / git log / TODO»):
самостоятельно выполни «восстановление контекста» ниже, кратко доложи «мы здесь»,
и **сразу берись за следующую задачу** из `docs/PROGRESS.md` по обычному циклу
(ровно одна задача → код+тесты → зелёный `pytest` → отметка `[x]` → обновление
`PROGRESS` → коммит). Не жди отдельного поручения на каждый шаг. Если сомнения нет —
делай; спрашивай, только если следующий шаг неоднозначен или задача затрагивает
необратимое/внешнее действие.

**В начале сессии (восстановление контекста):**
1. Прочитать `CLAUDE.md` (этот файл).
2. Прочитать `docs/PROGRESS.md` — текущая фаза, последняя/следующая задача.
3. `git log --oneline -15` — что менялось в последних сессиях.
4. Свериться с отметками `[x]` в `docs/TODO.md`.
5. **Назвать сессию по выполняемому шагу** — вызвать инструмент `mark_chapter`
   с `title` = имя текущей задачи из `docs/TODO.md`
   (напр. `feat(core): board wrapper`). Так отрезок работы в оглавлении сессии
   называется именем шага. Делать это сразу, как определена задача.
6. Кратко подтвердить пользователю «мы здесь, делаю задачу X».

**Во время сессии:**
- Брать ровно **одну** задачу из `docs/TODO.md`. Не смешивать задачи.
- Писать код **и тесты к нему вместе** (см. «Критерий готовности» ниже).

**В конце сессии (фиксация состояния):**
1. Прогнать `pytest` — **все тесты зелёные** (см. «Критерий готовности»).
2. Отметить `[x]` выполненную задачу в `docs/TODO.md`.
3. Обновить `docs/PROGRESS.md`: текущее состояние + строка в журнале сессий.
4. Сделать коммит `type(scope): summary` (одна задача = один коммит).
5. Если менялись решения — дописать в `docs/DECISIONS.md`.

## Критерий готовности задачи (Definition of Done)

Задача считается выполненной (и только тогда коммитится), когда выполнено ВСЁ:

1. **Код написан** и решает ровно то, что описано в пункте `docs/TODO.md`.
2. **Тесты написаны/обновлены** для нового кода:
   - юнит-тесты для `config`, `core`, `prompts`, `storage`, `analysis`, парсинга
     ходов и PGN — **обязательны**;
   - провайдеры — тесты на **моках** транспорта; реальные сетевые вызовы — отдельно
     и под `@pytest.mark`/`skip` без ключа;
   - движок (Stockfish) — тесты со `skip`, если бинарника нет;
   - веб — как минимум smoke-тест роутов/эндпоинтов.
3. **`pytest` прогнан и зелёный** (включая старые тесты — нет регрессий).
4. **Код проверен по тестам** — поведение подтверждено, а не только «написалось».
5. Документация обновлена, если задача меняет публичное поведение/форматы.

Отдельные `test(...)` пункты в `docs/TODO.md` — это места, где тесты особенно
объёмны и выносятся в свой коммит. Их наличие **не отменяет** правило: тесты пишутся
для каждой задачи с кодом, даже если отдельного `test(...)` пункта рядом нет.

## Правила работы в репозитории

- **Секреты только в `.env`.** Никогда не коммить ключи, не писать их в логи,
  `game.json`, PGN или отчёт. `.env` и `games/` — в `.gitignore`.
- **Источник истины по партии — `game.json`.** PGN и HTML генерируются из него.
- **Валидация ходов — через `python-chess`**, не вручную.
- **Каждая задача из `docs/TODO.md` = один коммит.** Не смешивай задачи.
- **Тесты обязательны для каждой задачи с кодом** — детали в разделе
  «Критерий готовности задачи (Definition of Done)» выше. Без зелёного `pytest`
  задача не коммитится.
- Код пиши в стиле окружающего кода; новые модули — с короткими docstring.

## Текущий статус

Единственный источник истины по «где мы сейчас» и «что дальше» — `docs/PROGRESS.md`
(здесь его НЕ дублируем, чтобы не расходилось). Этот файл описывает **правила и
ритуал**, а не текущий шаг. Последнюю/следующую задачу всегда бери из `PROGRESS.md`.
